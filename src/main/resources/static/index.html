CORE DEMONSTRATION

Complete customer ecosystem
in a single request

This is what makes GraphQL transformative for relational domains. One HTTP request. Full type-safe navigation across five tables and four relationships.

THE KILLER QUERY
1 roundtrip
query FullRelationalHierarchy {
  customers(
    filter: { entityNameContains: "Acme" }
    page: 0
    size: 1
  ) {
    content {
      id
      entityName
      type
      contacts {
        id firstName lastName email phone
      }
      shipmentLocations {
        id name addressLine1 addressLine2 city state zip locationInstructions gateCode
      }
      shipments {
        id activity itemDescription weight dimensions
        contact { id firstName lastName email }
        shipmentLocation { id name city state zip gateCode }
        serviceOffering { id description typeCd }
      }
    }
    totalElements
    totalPages
  }
}
Traverses all 5 entities with rich filtering and selection
Why this is impressive
Traditional REST architecture would require at least 5 separate HTTP calls plus client-side assembly logic.
Client controls exactly which fields and nested relations are returned — no over- or under-fetching.
Full referential integrity and rich domain model preserved in the API layer.
This pattern scales to any depth of relational complexity while remaining highly performant.
WRITE OPERATIONS

Insert & update data
with first-class mutations

GraphQL mutations are typed, validated through the domain, and persist via the same Spring Data JPA services used for reads. Create new customers or shipments, update names and details — then immediately re-query the graph to see the changes.

MUTATION Persisted to Derby
Create Shipment
mutation CreateShipment {
  # Inline for instant "Play" demo. In real usage prefer:
  # mutation CreateShipment($input: CreateShipmentInput!) { createShipment(input:$input){...} }
  createShipment(input: {
    customerId: 1
    serviceOfferingId: 4
    itemDescription: "Urgent prototype parts - inserted via GraphQL demo"
    weight: 14.25
    dimensions: "20x14x9"
    activity: DELIVERY
  }) {
    id itemDescription weight dimensions activity
    customer { id entityName }
    serviceOffering { description typeCd }
  }
}
Data-driven: first run a customers(filter: {entityNameContains: "Acme"}) query to obtain a real id, then pass it in $input.customerId. The response includes the new shipment plus its related objects.
MUTATION Persisted to Derby
Create New Customer
mutation CreateNewCustomer {
  createCustomer(input: {
    entityName: "Quantum Logistics Demo LLC"
    type: BUSINESS
  }) {
    id
    entityName
    type
    contacts { firstName }
    shipments { id itemDescription }
  }
}
Inserts a root Customer entity. The new record is immediately available for adding contacts, locations, and shipments in follow-up mutations or the hierarchy queries.
MUTATION Persisted to Derby
Update Existing Customer
mutation UpdateCustomerName {
  # Discover target via data-driven filter first:
  # customers(filter:{entityNameContains:"Acme"}){content{id}}
  updateCustomer(
    id: 1
    entityName: "Acme Manufacturing Inc. — Updated Live!"
    type: BUSINESS
  ) {
    id entityName type
    contacts { firstName lastName email }
  }
}
Target the customer using an ID obtained from a prior read (never hard-coded in production flows). Re-run any hierarchy query afterward to observe the change reflected across the graph.
After running a mutation, switch back to a read query (or re-run the hierarchy button) to see your changes in the relational results. All writes go through the same services, repositories, and transaction boundaries as the read paths.
MORE EXAMPLES

Filtering, Pagination,
Sorting & Reference Data

The same flexible API powers many other common operations. Client-controlled pagination, rich filtering, and sorting work at any level — including inside nested relations.

QUERY Filter + Page + Sort
Advanced Shipment Filtering
query AdvancedFiltering {
  shipments(
    filter: { activity: PICKUP, minWeight: 10 }
    page: 0
    size: 5
    sort: ["-weight"]
  ) {
    content {
      itemDescription weight activity
      customer { entityName }
    }
    totalElements totalPages
  }
}
Powerful server-side filtering, pagination and sorting on any field (including relations). No over-fetching.
QUERY Nested + Page
Nested Pagination
query NestedPagination {
  customers(
    filter: { entityNameContains: "Acme" }
    page: 0 size: 1
  ) {
    content {
      entityName
      shipments(
        page: 0 size: 2
        sort: ["-weight"]
      ) {
        weight itemDescription
        serviceOffering { description }
      }
    }
  }
}
Locate a customer with a filter, then ask for only a page of its shipments (with their own pagination & sort args).
QUERY Reference Data

query ListServiceOfferings {

Reference data used by createShipment mutations. Run this first in GraphiQL to pick real IDs for demo mutations.
QUERY List + Pagination
Get All Customers
query GetAllCustomers {
  customers(page: 0, size: 20) {
    content {
      id entityName type
    }
    totalElements
    totalPages
  }
}
Basic paginated list of all customers. Use page/size to control how many you get. totalElements gives the count.
QUERY List + Pagination
Get All Shipments
query GetAllShipments {
  shipments(page: 0, size: 20) {
    content {
      id itemDescription weight activity
      customer { entityName }
    }
    totalElements
    totalPages
  }
}
Paginated list of every shipment across all customers. Shows how the API supports broad listing with rich projections.
QUERY Aggregate / Count
Get Shipment Count
query ShipmentCount {
  shipments(page: 0, size: 1) {
    totalElements
  }
}
Efficient way to get just the total count without loading the actual shipment data. Uses the Page metadata.
THE DOMAIN MODEL

Five entities. Complete referential integrity.

Customer
id, entityName, type (BUSINESS/PERSONAL)
Root entity
Contact
firstName, lastName, phone, email
Many-to-one → Customer
ShipmentLocation
name, address*, city, state, zip, instructions, gateCode
Many-to-one → Customer
Shipment
itemDescription, weight, dimensions, activity (PICKUP/DELIVERY)
FKs to Customer, Contact, Location, ServiceOffering
ServiceOffering
description, typeCd
Referenced by Shipment
All relationships are properly modeled with JPA and exposed through GraphQL with rich selection.
